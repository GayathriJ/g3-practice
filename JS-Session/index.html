5. Where do you place javascript in HTML? What is the correct position?

Javascript should be placed at the bottom of the page. Before the closing tag of body. Because whenever browser sees a script tag, it pauses parsing to execute the script first. This will lead to page load delay. This solution works for both old and modern browsers.
Alternatively in modern browsers, defer (<scrip defer>) can be used in the script tag which doesn't pause the parser but waits for the parser to complete and then executes the script. Also using defer ensures that the scripts will be executed in the order given in the page. This becomes useful when handling jquery, where other scripts might need the jquery js file first.
Another option is to use async (<script async>). This is similar to defer, but it doesn’t ensure the order of execution but executes any script thats available after the page is parsed.

children()– search through the matched elements’ child only (single level down).

    • 28. Explain bind() vs live() vs delegate() vs on() methods.

    Using click as example event below
    Bind() :
    $( "a" ).bind( “click”, function(){}) ;
    Shortcut of this is $(“a”).click(function(){});
    In bind, the event handler is attached to all existing links (in this case) that are currently present in the DOM. This uses more memory . Also, any links added dynamically in the future will not have this handler attached.
    Opposite is $(“a”).unbind();
    live() :
    $( "a" ).live( “click”, function(){});
    In this the handler is delegated to document. So, any dynamically added new links will also work with the event handler. But because it delegates the handler to the document, it affects the performance. Also chaining is not properly supported in this. Live and die are removed in jquery 1.9 and should not be used.
    Opposite is $(“a”).die();
    delegate():
    $( "table" ).delegate( "td", "click", function() {});
    In this, unlike live, the handler is delegated to the selector (table). So it doesnt affect performance. So in this case, any existing or new td of the table will have the event handler working for it. Also it supports chaining.
    Opposite is $(“table”)undelegate();
    on():
    $( "#dataTable tbody tr" ).on( "click", function() { }); // acts like bind
    $( "#dataTable tbody" ).on( "click", "tr", function() { }); // acts like delegate
    on is the recommended one introduced in 1.7 which is intended to replace all the above.
    Opposite is $(“table”).off();
    Note: In all the opposites given, event type and selector can be given to specifically remove the event handler of some events / selectors alone.

    31. What is event bubbling and capturing in javascript?

    Consider the below html which has one span nested inside the div.
    <div>
    DIV
    <span>
    Span
    </span>
    <div>
    Now event handlers are attached to both span and div.
    $('div').click(function() { alert('div clicked'); });
    $('span').click(function() { alert('span clicked'); });
    When the text Span is clicked on the page, first the event handler of span will be called , then the event handler of div will be called. This is called event bubbling. That is, once the handler on the child is triggered, the handler of the parent is also triggered. (This is applicable for ancestors as well, not just for parents, if they also have event handlers attached to them)
    The opposite of this event capturing (trickling down). In that case when span is clicked, first div's handler is called and then span's handler is called. Browsers dont support this by default. Only by using addEventListener(event, callback, true) this behaviour can be replicated.

    32. What is event stopPropagation in jquery

    To prevent bubbling up of events, stopPropagation can be used.
    e.g.
    $('div').click(function() { alert('div clicked'); });
    $('span').click(function(e) { alert('span clicked'); e.stopPropagation(); });
    With respect to the previous example, by adding the e.stopPropagation, after the span's hanlder is called, it will stop and not call the div's handler.

    33. What is the difference between trigger and triggerHandler in jquery ?

    trigger and triggerHandler are used for triggering an event without the user need to do it via the user interface.
    $('#clickMe').trigger('click');
    $('#clickMe').triggerHandler('click');
    The differences between them are that, triggerHandler prevents bubbling up of events.
    TriggerHandler works on only the first matched selector, whereas trigger works on all. TriggerHandler prevents the default action of the browser on some events such as select and focus (as can be seen in the style change).
    Trigger returns jquery object allowing chaining. But triggerHandler returns the whatever value was returned by the last handler it caused to be executed. If no handlers are triggered, it returns undefined

Only $(“input”).prop(“checkbox”) will return true or false depending on whether is the checkbox is checked or not.

    37. Explain .empty() vs .remove() vs .detach()

    Empty() → This is used to remove all the content/text inside the selected element and the descendants of the selected element. But it keeps the tag intact.
    Remove() → This removes the selected element itself along with its children. It also removes any events added to via jquery and any jquery data (added using data()). Its faster than detach.
    detach() → detach is similar to remove. But it retains the event information and the jquery data. So the element can be reintroduced again without much issues.

    43. How to invoke https using ajax. Explain about cors and jsonp.

    Cross-origin resource sharing (CORS) is a mechanism that allows restricted resources (e.g. fonts, JavaScript, etc.) on a web page to be requested from another domain outside the domain from which the resource originated.

    When a CORS-compatible browser attempts to make a cross-origin request:

    The browser sends the request with an Origin HTTP header. The value of this header is the domain that served the parent page. When a page from http://www.foo.com attempts to access a user's data in bar.com, the following request header would be sent to bar.com:
    Origin: http://www.foo.com
    The server may respond with:
    An Access-Control-Allow-Origin (ACAO) header in its response indicating which origin sites are allowed. For example:
    Access-Control-Allow-Origin: http://www.foo.com
    An error page if the server does not allow the cross-origin request
    An Access-Control-Allow-Origin (ACAO) header with a wildcard that allows all domains:
    Access-Control-Allow-Origin: *

    The HTTP headers that relate to CORS are:

    Request headers
    Origin
    Access-Control-Request-Method
    Access-Control-Request-Headers
    Response headers
    Access-Control-Allow-Origin
    Access-Control-Allow-Credentials
    Access-Control-Expose-Headers
    Access-Control-Max-Age
    Access-Control-Allow-Methods
    Access-Control-Allow-Headers

    JSONP or “JSON with padding” is a communication technique used in JavaScript programs running in web browsers to request data from a server in a different domain, something prohibited by typical web browsers because of the same-origin policy. JSONP takes advantage of the fact that browsers do not enforce the same-origin policy on <script> tags. Since it works through <script> tags, JSONP supports only the GET request method.

    Note that for JSONP to work, a server must know how to reply with JSONP-formatted results. JSONP does not work with JSON-formatted results. The JSONP parameters passed as arguments to a script are defined by the server. A typical JSONP request is similar to the following sample code:

    // Request sent via a script tag
    <script src="https://status.github.com/api/status.json?callback=apiStatus"></script>
    // Data received as an execution of the predefined function.
    <script> function apiStatus(data) { console.log(data.status); } </script>

    or

    <script type="application/javascript"
    src="http://server.example.com/Users/1234?jsonp=parseResponse">
    </script>

    In this example, the received payload would be:

    parseResponse({"Name": "Foo", "Id": 1234, "Rank": 7});

    CORS can be used as a modern alternative to the JSONP pattern. While JSONP supports only the GET request method, CORS also supports other types of HTTP requests. Using CORS enables a web programmer to use regular XMLHttpRequest, which supports better error handling than JSONP. On the other hand, JSONP works on legacy browsers which predate CORS support. CORS is supported by most modern web browsers.

    45. Difference between data type and content type in ajax.

    contentType (default: 'application/x-www-form-urlencoded; charset=UTF-8')

    Type: String

    When sending data to the server, use this content type. Default is "application/x-www-form-urlencoded; charset=UTF-8", which is fine for most cases. If you explicitly pass in a content-type to $.ajax(), then it'll always be sent to the server (even if no data is sent). If no charset is specified, data will be transmitted to the server using the server's default charset; you must decode this appropriately on the server side.

    dataType (default: Intelligent Guess (xml, json, script, or html))

    Type: String
    The type of data that you're expecting back from the server. If none is specified, jQuery will try to infer it based on the MIME type of the response (an XML MIME type will yield XML, in 1.4 JSON will yield a JavaScript object, in 1.4 script will execute the script, and anything else will be returned as a string).
